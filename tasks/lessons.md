# レッスン記録 (MCPサーバー統合)

## 課題と解決策

### 1. BlenderのAPIスレッド制限
- **課題**: Pythonの標準のソケットサーバー (`socket.accept()`) やマルチスレッドを使用して非同期にコマンドを受信しようとすると、受信した別スレッドから `bpy.ops` 等のBlender APIを呼び出そうとしたときにクラッシュしたり、メインスレッド以外からのAPI操作としてブロックされる事象が発生します。
- **解決策**: 別スレッド (`_server_loop`) ではコマンド（JSON）を受信して `queue.Queue` に放り込む処理のみ行います。アドオンの `start_server()` にて `bpy.app.timers.register` を用いて、定期的に `process_queue` をメインスレッドで実行させるようにしました。これにより、1件ずつ着実にメインスレッドのキューから取り出され、安全にAPI操作を行える環境を構築できました。

### 2. アドオンへのモジュール組み込み構造
- **課題**: 複数ファイル（`gpurental_addon.py` と `blender_mcp_server.py`）で構成される際、パッケージ機能を用いずに1階層のスクリプトとして扱うにはモジュールパスが認識されない場合があります。
- **解決策**: `import sys, os` でパスを無理やり操作する代わりに、BlenderのユーザーScriptsディレクトリで同列にあることを活かし、単に `try: import blender_mcp_server except ImportError: None` というセーフフォールバック付きのインポート処理を加え、存在する場合のみUIを表示してトグルできるようにしました（疎結合な設計を実現）。

### 3. ソケットの確実なクローズ処理
- **課題**: アドオンの再読み込みや機能をオフにする際に、ソケットサーバーのポート（8123）が「アドレス使用中 (Address Already in Use)」でエラーになるリスクがあります。
- **解決策**: サーバー立ち上げ時に `setsockopt(socket.SOL_SOCKET, socket.SO_REUSEADDR, 1)` を付与し、さらに `socket.settimeout(1.0)` として受信待機中に定期的に `_running` フラグをチェックして安全に `close()` 終了できる構造を採用しました。
